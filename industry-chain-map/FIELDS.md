# FIELDS.md 字段总账

## 总规则

本项目每个字段都同时存在于五处：① 数据 JSON（`/data`）、② schema 定义、③ 数据提取模板、④ 推导/计算规则、⑤ 界面渲染（节点缩略 + 详情面板）。任何字段的增删改，都必须五处同步更新，并同步更新本文档。

本文档是全项目字段定义和缺口评分规则的唯一权威来源。`gap-rules` 等代码必须严格按本文档实现；以后修改本文档中的渗透率、库存周期、产能利用率等规则后，代码、数据模板、schema 和界面说明都必须同步改到一致。

新增产业链只是在现有字段 schema 下填入新的产业数据，不需要逐个产业登记进本文档。本文档只记录字段定义、单位、呈现方式、推导规则和通用流程，不记录具体产业的数据内容。

## 字段登记格式

每个字段条目按以下口径理解：

- 数据 JSON：字段在 `/data/*.json`、`/data/index.json` 或 `/data/config.json` 中的位置。
- schema 定义：字段在项目 schema 中应如何约束。当前项目尚无独立 schema 文件，本文档先作为 schema 的权威来源。
- 数据提取模板：字段在数据采集和更新模板中应如何填写。
- 推导/计算规则：字段是否由程序推导、是否参与评分、阈值如何定义。
- 界面渲染：字段在中间节点缩略和右侧详情面板中的呈现方式。

## 节点基础字段

### `id` / 环节唯一标识

- 含义说明：产业链环节的稳定英文标识，用于树节点定位、引用、搜索和跨文件复用。
- 数据类型与单位：字符串；无单位。建议使用小写英文、数字、下划线或短横线，必须在同一展开后的产业链树内唯一。
- 数据 JSON：位于每个节点对象的 `id`。`data/index.json.chains[].id` 是产业链入口 id。
- schema 定义：节点必填；`ref` 引用节点解析后也必须拥有 `id`。
- 数据提取模板：每个环节必须填写；同一通用环节跨产业复用时保持同一 id。
- 推导/计算规则：不参与缺口评分；`loader` 和树渲染依赖它建立节点身份。
- 界面渲染：详情面板标题区显示为 `ID`；节点搜索匹配 `id`。

### `name` / 环节中文名

- 含义说明：产业链环节在界面上的中文显示名。
- 数据类型与单位：字符串；无单位。
- 数据 JSON：位于每个节点对象的 `name`；`data/index.json.chains[].name` 是产业链入口名称。
- schema 定义：节点必填；索引中的产业链条目必填。
- 数据提取模板：使用行业通用、简洁、可读的中文名称。
- 推导/计算规则：不参与评分。
- 界面渲染：中间画布节点主标题显示；详情面板标题显示；上游节点表显示。

### `layer_updated` / 环节更新时间

- 含义说明：该节点数据最近一次更新的年月或日期。
- 数据类型与单位：字符串；推荐 `YYYY-MM`，必要时可用 `YYYY-MM-DD`。
- 数据 JSON：位于每个节点对象的 `layer_updated`。
- schema 定义：节点推荐填写；缺失时界面显示 `-`。
- 数据提取模板：每次更新节点数据时同步更新。
- 推导/计算规则：不参与评分。
- 界面渲染：详情面板底部显示为最后更新时间。

### `upstream` / 上游环节数组

- 含义说明：当前节点的上游节点列表，构成产业链树结构。
- 数据类型与单位：数组；元素为完整节点对象或包含 `ref` 的引用对象。
- 数据 JSON：位于每个节点对象的 `upstream`。
- schema 定义：节点可选；缺失或空数组表示无上游展开节点。
- 数据提取模板：按产业链上下游关系填写；已有通用环节优先使用 `ref` 复用。
- 推导/计算规则：不参与缺口评分；树渲染用它作为子节点来源。
- 界面渲染：中间画布按 `upstream` 生成可展开树；详情面板上游节点表显示 `name`、`id`、`ref`。

### `ref` / 跨文件引用

- 含义说明：引用已有节点定义，避免在多个产业链中重复维护同一环节。
- 数据类型与单位：字符串；无单位。
- 数据 JSON：常见于 `upstream[]` 元素，例如引用另一个数据文件中的节点。
- schema 定义：引用对象可只含 `ref`，加载阶段应解析为完整节点；解析失败应报错。
- 数据提取模板：新增产业链时，若某上游环节已有权威定义，应优先填写 `ref` 而不是复制完整节点。
- 推导/计算规则：不参与评分；加载器解析引用后，后续计算和渲染使用解析后的完整节点。
- 界面渲染：详情面板上游节点表显示 `ref`；节点缩略不直接显示。

## 指标字段

### `metrics.market_size` / 市场规模

- 含义说明：该环节对应市场的规模信息。
- 数据类型与单位：对象；由 `value`、`unit`、`year`、`note` 组成。
- 数据 JSON：位于 `metrics.market_size`。
- schema 定义：推荐字段。`value` 为数字或可核验的短文本，`unit` 为字符串，`year` 为年份数字或字符串，`note` 为补充说明。
- 数据提取模板：填写最新可核验口径；必须注明单位和年份，口径差异写入 `note`。
- 推导/计算规则：不参与缺口评分。
- 界面渲染：详情面板顶部指标卡显示；节点缩略当前不显示。

#### `metrics.market_size.value` / 市场规模数值

- 含义说明：市场规模的主数值。
- 数据类型与单位：数字优先；单位由 `unit` 提供。
- 五处定义/呈现：数据中作为原始值；schema 约束为数字优先；模板要求采集值；无推导；详情指标卡拼接展示。

#### `metrics.market_size.unit` / 市场规模单位

- 含义说明：市场规模的计量单位，例如 `亿元`、`亿美元`。
- 数据类型与单位：字符串；字段自身无单位。
- 五处定义/呈现：数据中与 `value` 同时维护；schema 约束为字符串；模板要求必填；无推导；详情指标卡紧跟数值展示。

#### `metrics.market_size.year` / 市场规模年份

- 含义说明：市场规模对应的统计年份。
- 数据类型与单位：数字或字符串；年份。
- 五处定义/呈现：数据中记录口径年份；schema 推荐 `YYYY`；模板要求注明；无推导；详情指标卡展示。

#### `metrics.market_size.note` / 市场规模说明

- 含义说明：市场规模口径、来源、边界或估算说明。
- 数据类型与单位：字符串；无单位。
- 五处定义/呈现：数据中可选；schema 允许为空；模板用于记录口径差异；无推导；详情指标卡展示。

### `metrics.cagr` / CAGR

- 含义说明：该环节市场规模或相关指标在指定周期内的复合增速。
- 数据类型与单位：对象；由 `value`、`unit`、`period`、`note` 组成。
- 数据 JSON：位于 `metrics.cagr`。
- schema 定义：推荐字段。`value` 为数字优先，`unit` 通常为 `%`，`period` 为时间区间字符串。
- 数据提取模板：必须注明预测或历史周期；口径写入 `note`。
- 推导/计算规则：当前不由程序计算，作为采集字段录入；不参与缺口评分。
- 界面渲染：详情面板顶部指标卡显示；节点缩略当前不显示。

#### `metrics.cagr.value` / CAGR 数值

- 含义说明：复合增速数值。
- 数据类型与单位：数字优先；单位由 `unit` 提供，通常为 `%`。
- 五处定义/呈现：数据中作为原始值；schema 约束为数字优先；模板要求采集；当前无程序推导；详情指标卡拼接展示。

#### `metrics.cagr.unit` / CAGR 单位

- 含义说明：CAGR 的单位。
- 数据类型与单位：字符串；通常为 `%`。
- 五处定义/呈现：数据中维护；schema 推荐 `%`；模板要求注明；无推导；详情指标卡展示。

#### `metrics.cagr.period` / CAGR 周期

- 含义说明：CAGR 对应的统计或预测周期。
- 数据类型与单位：字符串；例如 `2024-2028E`。
- 五处定义/呈现：数据中维护；schema 约束为字符串；模板要求注明；无推导；详情指标卡展示。

#### `metrics.cagr.note` / CAGR 说明

- 含义说明：CAGR 口径、来源、是否预测值等说明。
- 数据类型与单位：字符串；无单位。
- 五处定义/呈现：数据中可选；schema 允许为空；模板记录来源口径；无推导；详情指标卡展示。

### `metrics.value_share` / 价值量占比

## 缺口字段与评分规则

### `gap.status` / 缺口状态

- 含义说明：当前环节供需状态的汇总判断。
- 数据类型与单位：枚举字符串；允许值为 `tight`（紧缺）、`balance`（平衡）、`surplus`（过剩）。
- 数据 JSON：位于 `gap.status`。权威规则要求该字段应由程序根据底层指标推导；源数据中原则上不手填，必要暂存时必须与推导结果一致。
- schema 定义：运行后必有；源数据中应允许为空或缺失。
- 数据提取模板：只采集底层指标和 `gap.reasoning`，不以人工结论替代规则推导。
- 推导/计算规则：由 `gap.indicators` 四项指标逐项映射为 `+1/0/-1` 后求和，按总分映射状态。
- 界面渲染：节点缩略显示状态；详情面板顶部指标卡和缺口状态 section 显示状态、分数和解释。界面应显示中文标签或中英一致标签。

### `gap.score` / 缺口总分

- 含义说明：缺口底层指标评分求和后的运行时总分。
- 数据类型与单位：数字；无单位。
- 数据 JSON：不应写入源数据文件；由运行时生成。
- schema 定义：运行时字段；源数据 schema 中应禁止手填或标记为只读派生字段。
- 数据提取模板：不填写。
- 推导/计算规则：`capacity_utilization`、`inventory_cycle`、`penetration_rate`、`supply_tightness` 四项分数相加。
- 界面渲染：详情面板缺口指标卡显示为 `status / score N`；缺口明细 section 可辅助显示。

### `gap.breakdown` / 缺口评分明细

- 含义说明：每个底层指标的原始值和评分结果。
- 数据类型与单位：数组；元素包含 `label`、`value`、`score`。
- 数据 JSON：不应写入源数据文件；由运行时生成。
- schema 定义：运行时字段；源数据 schema 中应禁止手填或标记为只读派生字段。
- 数据提取模板：不填写。
- 推导/计算规则：按本文档的四项底层指标规则生成。
- 界面渲染：详情面板缺口状态 section 逐行显示指标名、原始值和分数。

### `gap.reasoning` / 缺口判断理由

- 含义说明：对缺口状态的文字解释，说明供需、库存、渗透率、产能利用率等主要依据。
- 数据类型与单位：字符串；无单位。
- 数据 JSON：位于 `gap.reasoning`。
- schema 定义：推荐填写；允许为空。
- 数据提取模板：用简洁研报语言说明判断依据，不替代底层指标。
- 推导/计算规则：不参与评分；可用于人工审阅和详情展示。
- 界面渲染：详情面板缺口状态 section 展示；节点缩略不显示。

### `gap.indicators.capacity_utilization` / 产能利用率

- 含义说明：行业或该环节代表性产能利用水平，用于判断供给紧张程度。
- 数据类型与单位：数字；单位为 `%`，数据中只填数值，不写 `%`。
- 数据 JSON：位于 `gap.indicators.capacity_utilization`。
- schema 定义：数字或 `null`；推荐使用数字。字符串描述不应作为可评分值。
- 数据提取模板：填写可比较的百分比数值；如果只有描述，先转化为可评分标准表述或留空并在 `reasoning` 中说明。
- 推导/计算规则：
  - `value > 85`：`+1`，表示产能利用率高，供给偏紧。
  - `70 <= value <= 85`：`0`，表示中性或平衡。
  - `value < 70`：`-1`，表示产能利用率偏低，供给偏松。
  - 缺失或无法解析：`0`，但应在数据审阅中标记为低置信度。
- 界面渲染：详情面板缺口评分明细显示原始值和分数；节点缩略不单独显示。

### `gap.indicators.inventory_cycle` / 库存周期

- 含义说明：库存所处阶段，用于判断需求和补库周期位置。
- 数据类型与单位：枚举字符串；无单位。
- 数据 JSON：位于 `gap.indicators.inventory_cycle`。
- schema 定义：必须使用标准可评分表述之一，避免长句直接入库。
- 数据提取模板：从研报描述中归一为标准表述；详细解释写入 `gap.reasoning`。
- 推导/计算规则：
  - `去库尾声`：`+1`
  - `底部回补`：`+1`
  - `底部回补/去库尾声`：`+1`
  - `主动补库`：`0`
  - `被动累库`：`-1`
  - `主动去库`：`-1`
  - 其他值或缺失：`0`，但应列为需要标准化的数据问题。
- 界面渲染：详情面板缺口评分明细显示原始值和分数；节点缩略不单独显示。

### `gap.indicators.penetration_rate` / 渗透率

- 含义说明：新技术、新材料或新工艺在目标市场中的渗透水平，用于判断成长阶段和潜在紧缺弹性。
- 数据类型与单位：数字；单位为 `%`，数据中只填数值，不写 `%`。
- 数据 JSON：位于 `gap.indicators.penetration_rate`。
- schema 定义：数字或 `null`；推荐使用数字。字符串描述不应作为可评分值。
- 数据提取模板：填写可比较的百分比数值；若没有明确数值，可在 `reasoning` 中描述，但不作为可评分字段。
- 推导/计算规则：
  - `value < 30`：`+1`，低渗透，成长早期，供给可能偏紧。
  - `30 <= value <= 60`：`0`，中性。
  - `value > 60`：`-1`，高渗透，成熟度较高，紧缺弹性降低。
  - 缺失或无法解析：`0`，但应在数据审阅中标记为低置信度。
- 界面渲染：详情面板缺口评分明细显示原始值和分数；节点缩略不单独显示。

### `gap.indicators.supply_tightness` / 供需紧张度

- 含义说明：对当前供需关系的归一化描述。
- 数据类型与单位：枚举字符串；允许值为 `偏紧`、`平衡`、`偏松`。
- 数据 JSON：位于 `gap.indicators.supply_tightness`。
- schema 定义：必须使用标准枚举值；不使用长句。
- 数据提取模板：将研报中的供需描述归一为三个标准值；证据写入 `gap.reasoning`。
- 推导/计算规则：
  - `偏紧`：`+1`
  - `平衡`：`0`
  - `偏松`：`-1`
  - 其他值或缺失：`0`，但应列为需要标准化的数据问题。
- 界面渲染：详情面板缺口评分明细显示原始值和分数；节点缩略不单独显示。

### 缺口总分到状态映射

- 计算公式：`gap.score = capacity_utilization_score + inventory_cycle_score + penetration_rate_score + supply_tightness_score`。
- `gap.score >= +2`：`gap.status = tight`，中文显示为“紧缺”。
- `-1 <= gap.score <= +1`：`gap.status = balance`，中文显示为“平衡”。
- `gap.score <= -2`：`gap.status = surplus`，中文显示为“过剩”。
- 若源数据临时存在人工填写的 `gap.status`，也必须与上述推导结果一致；后续代码应以本文档规则为准，避免人工状态覆盖推导状态。

## 公司字段

## 新增产业链的标准流程（SOP）

新增产业链是按现有字段 schema 填写新的产业数据，不需要把每个产业逐个登记进本文档。本文档只维护字段定义、字段规则和通用流程。

1. 新建产业链数据文件：在 `/data` 下创建新的 `{chain_id}.json`，根节点必须包含 `id`、`name`、`metrics`、`gap`、`companies`、`tech_evolution`、`upstream`、`layer_updated` 等现有字段结构。
2. 复用已有环节：如果新增产业链中的上游或共用环节已在其他数据文件中存在，优先用 `ref` 引用已有环节，避免复制粘贴造成多处不一致。
3. 更新索引：在 `/data/index.json` 的 `chains` 中新增 `{ id, name, note }`；如需默认打开该产业链，再更新 `default_chain`。
4. 按总账 schema 填写字段：所有字段的类型、单位、必填/可选口径以本文档为准。市场规模必须带单位和年份，CAGR 必须带周期，价值量占比必须带参照对象。
5. 使用标准缺口指标：`capacity_utilization` 和 `penetration_rate` 使用百分比数值；`inventory_cycle` 只能使用总账中列出的标准表述；`supply_tightness` 只能使用 `偏紧`、`平衡`、`偏松`。
6. 不手填运行时字段：不要在源数据中填写 `gap.score` 和 `gap.breakdown`。`gap.status` 原则上由规则推导，若临时保留人工状态，必须与规则推导结果一致。
7. 写清判断理由：无法标准化为底层指标的研报描述，写入 `gap.reasoning` 或指标 `note` 类字段，不要塞进可评分字段。
8. 更新五处来源：如果新增产业链过程中发现需要新增字段，必须同步更新 `/data`、schema、数据提取模板、推导/计算规则、界面渲染，并更新本文档。
9. 完成校验：新增后检查 JSON 可加载、节点可展开、详情面板字段完整、缺口评分明细符合本文档规则。

## 当前五处定义不一致清单

以下问题是截至当前代码和数据状态发现的不一致，后续同步代码和模板时应逐项确认处理。

1. 独立 schema 文件缺失：当前没有单独的 JSON Schema 或等价 schema 文件，字段约束分散在需求规格书、数据提取模板和代码中。本文档先作为权威 schema 来源。
2. `gap.status` 的来源冲突：规格和模板要求缺口状态由程序推导，但源数据中已存在手工填写的 `gap.status`，例如 `data/pcb.json` 根节点为 `tight`。当前 `app.js` 只在缺失时写入推导状态，会保留人工状态。
3. 描述型指标无法稳定评分：模板允许 `penetration_rate`、`capacity_utilization` 为“数值或描述”，但当前 `gap-rules.js` 实际按数值阈值比较；字符串描述会导致评分不可控或退化为中性。
4. 库存周期长描述未标准化：当前评分代码只精确匹配配置中的标准字符串。数据或文档中的长描述，如“高景气/订单能见度8-10个月”“随PCB高端扩产景气上行”“紧缺/补库”等，不在标准集合中，会被算作 `0`。
5. `value_share.relative_to` 未明确渲染：数据和模板中有占比参照对象，但当前详情面板 `formatMetric` 未专门展示 `relative_to`，容易丢失口径。
6. `config.ref_map` 未使用：`data/config.json.ref_map` 存在，但当前未发现加载或渲染代码使用它。
7. 缺口状态中英文展示不统一：节点缩略和详情面板当前倾向显示原始英文 `tight`、`balance`、`surplus`，而业务说明和图例需要中文“紧缺/平衡/过剩”。
8. 运行时字段缺少源数据禁填说明：`gap.score` 和 `gap.breakdown` 会被 UI 使用，但数据模板和 schema 还没有明确标注它们是运行时字段、源数据不应手填。
9. 代码中文标签存在编码异常：当前部分 JS 文件中的中文 UI 文案和缺口明细 label 出现乱码，可能影响字段展示和后续人工核对。

## 代码同步要求

- `gap-rules.js` 必须以本文档的缺口评分规则为准；默认阈值、配置阈值和文档阈值不得出现三套口径。
- 数据提取模板必须要求库存周期和供需紧张度使用本文档标准枚举。
